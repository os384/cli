## What evolved

The initial fix (making `384 env` generate correct env.js) was superseded by
a broader architecture discussion. Peter decided to split the monolithic env.js
into three files with clear separation of concerns:

1. **`env.js`** — one-liner: `const serverType = 'local';` (or 'dev')
   Written by the Makefile from `ENV=local|dev`. Not generated by CLI.
   Needed for static HTML pages that load config via `<script>` tags.

2. **`keys.js`** — pre-computed crypto handles in an IIFE that sets `globalThis.env`.
   Generated by `384 init` and written to `~/.os384/keys.js`.
   Symlinked into lib384/, services/, cli/ etc. via Makefile `env` target.
   Contains full ChannelHandle objects (channelId, ownerPublicKey derived from
   private keys via SB384 async crypto at init time).

3. **`config.js`** — unchanged profile selection logic.
   Reads `globalThis.env`, maps `configServerType` to flat `configuration` object.

## Discussion notes

- `config.js` is NOT AI slop — it's original wiring from the old codebase. All
  40+ tests import it. Left as-is.
- "mint" vs "token": `384 mint` already exists (creates a VALID authorized token).
  `384 token` creates a correctly-formed identifier only (not authorized).
- Peter explicitly rejected auto-detection defaults: "instead of being clever about
  default behaviors, might as well enforce that ENV must exist."
- The reason keys.js must pre-compute handles is that SB384 key derivation is async.
  Not all consumers (static HTML tests) can do async imports. Pre-computing at
  `384 init` time and emitting synchronous JS solves this.
- Future: separate `tests` repo in os384 GitHub org for integration tests (04+).
  lib384 keeps pure unit tests (01-03). Deno workspaces for source-level debugging.
